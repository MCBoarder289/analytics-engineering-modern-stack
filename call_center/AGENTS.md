# AGENTS.md — call_center (dbt Project)

This directory is the dbt transformation project. It is **not** a Python package and has no virtual environment of its own — dbt runs through the **repo root venv**.

## Running dbt Commands

Always run dbt from the **repo root** using `uv run`, passing `--project-dir` and `--profiles-dir`:

```bash
# From repo root
uv run dbt run     --project-dir call_center --profiles-dir call_center
uv run dbt test    --project-dir call_center --profiles-dir call_center
uv run dbt build   --project-dir call_center --profiles-dir call_center
uv run dbt compile --project-dir call_center --profiles-dir call_center
uv run dbt docs generate --project-dir call_center --profiles-dir call_center
```

### Model Selectors

```bash
# Run a specific layer
uv run dbt run --project-dir call_center --profiles-dir call_center --select staging
uv run dbt run --project-dir call_center --profiles-dir call_center --select data_marts
uv run dbt run --project-dir call_center --profiles-dir call_center --select ops_analysis

# Run a model and all its upstream dependencies
uv run dbt run --project-dir call_center --profiles-dir call_center --select +mart_surveys

# Run only a specific model
uv run dbt run --project-dir call_center --profiles-dir call_center --select stg_calls
```

## Project Structure

```
call_center/
├── dbt_project.yml         # Project config, schema mappings
├── profiles.yml            # Connection profile (generated by mds init-env — not committed as-is)
├── packages.yml            # dbt package dependencies
├── models/
│   ├── sources/            # Source declarations (raw ingested tables)
│   ├── staging/            # Light cleaning & standardization (+schema: staging)
│   ├── ops_analysis/       # Intermediate operational models (+schema: core)
│   ├── data_marts/         # Business-facing mart models (+schema: mart)
├── seeds/                  # Static CSV reference data (agents, managers, customers)
├── macros/                 # Reusable Jinja/SQL macros
├── tests/                  # Custom data quality tests
├── analyses/               # Ad-hoc analysis SQL (not materialized)
└── snapshots/              # dbt snapshot definitions
```

## Schema Layout

| Layer | dbt group | DuckDB schema |
|-------|-----------|--------------|
| Staging | `staging_data` | `staging` |
| Ops Analysis | `ops_analysis` | `core` |
| Data Marts | `data_marts` | `mart` |

## Connection Profile

`profiles.yml` is generated by `mds init-env` with absolute paths for your machine. The profile name is `call_center` and targets DuckDB at `data/warehouse_dev.duckdb` (relative to repo root).

If `profiles.yml` is missing or has wrong paths, regenerate it:

```bash
# From repo root
uv run mds init-env --no-prompt   # always use --no-prompt; bare init-env will block on interactive prompts
```

## Installing dbt Packages

```bash
uv run dbt deps --project-dir call_center --profiles-dir call_center
```

## Seeds

Seeds are static CSV reference tables. Load them with:

```bash
uv run dbt seed --project-dir call_center --profiles-dir call_center
```

In normal operation, seeds are loaded by Dagster as part of the pipeline. Only run `dbt seed` directly when working on the dbt project in isolation.

## Important Notes for Agents

- dbt runs via the **root venv** — always use `uv run dbt` from the repo root with explicit `--project-dir` and `--profiles-dir` flags.
- Do **not** `cd` into `call_center/` and run bare `dbt` commands unless you have activated the root venv manually.
- SQL files use DuckDB dialect. Avoid Snowflake/BigQuery/Redshift-specific syntax.
- The warehouse file is `data/warehouse_dev.duckdb` at the repo root. Dagster must have run at least the ingestion assets before dbt models can be built.
- `target/` and `dbt_packages/` are gitignored — don't commit them.
